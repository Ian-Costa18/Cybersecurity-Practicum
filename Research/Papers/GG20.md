# Paper

- Title: One Round Threshold ECDSA with Identifiable Abort
- Authors: Rosario Gennaro, Steven Goldfeder
- Year: 2020
- URL / DOI: https://ia.cr/2021/060

---

## Quick summary (1–2 lines)

- One-line summary: Protocol for threshold ECDSA signatures with one-round online signing and identifiable abort mechanism that allows efficient detection and attribution of misbehaving parties during signing.

## Why it matters

- Connection to our project (e.g., authentication, proxy, usability): Directly relevant to threshold signature schemes for distributed authentication and cryptocurrency multi-sig wallets. Addresses critical failure modes in distributed key management where malicious parties can cause protocol abort without identification.
- Key takeaway: Identifiable abort capability is essential for practical deployment of threshold cryptography, enabling committees to identify and remove bad actors without requiring retries, reducing DoS attack surface.

## What they did

- Methods / system / experiment (short bullets)
  - Presented two threshold ECDSA protocols with improved round complexity
  - Section 3: One-round online protocol with identifiable abort (main contribution)
  - Section 5: Simplified protocol tolerating anonymous aborts with reduced round complexity
  - Implemented protocols in Go using Binance's tss-lib as baseline
  - Benchmarked against GG18 (Gennaro-Goldfeder 2018) protocol
  - Employed multiplicative-to-additive (MtA) share conversion protocols
  - Used Feldman verifiable secret sharing (VSS) for consistency verification
  - Integrated zero-knowledge proofs for each phase to ensure correctness

- Main results / claims
  - Achieves single-round online phase for signing (after offline preprocessing)
  - Overall fewer rounds than GG18 even if entire protocol run online
  - Efficient identification of cheating parties in both key generation and signing phases
  - Protocol simultaneously reduces rounds, computation, and bandwidth vs GG18
  - Benchmark: for 10 participants, ~4289ms vs 3849ms for GG18 (roughly 10% overhead for identifiable abort)
  - Bandwidth efficiency comparable or better than GG18

## Challenges and solutions

- Challenge: Previous protocols unable to identify which party caused abort, especially in key generation phase
  - Solution: Employ a pre-processing phase with distributed verification check on shares k^-1 and kr before message is known. Once message known, can safely reveal shares of signature to identify bad players without leaking secret key.

- Challenge: Detecting misbehavior in signing protocol when multiple failure sources possible (Phase 5-7 abort types)
  - Solution: Force all parties to publish values ki, γi, αij, βij and intermediate computations in clear. Any player can reconstruct equations and identify which party violates constraints.

- Challenge: Simulation security for identifiable abort requires sophisticated handling of encryption commitments when adversary may try to "open" encryption differently
  - Solution: Carefully construct protocol so simulator can commit to randomizer R and ensure consistency without being "caught". Uses non-malleable commitments and explicit value binding.

- Challenge: Handling DoS via protocol abort in critical applications (e.g., Keep Network for blockchain atomic swaps)
  - Solution: Identifiable abort enables committee to punish only misbehaving member, lock up funds securely without requiring protocol restart.

## Limitations / open questions

- Limitations noticed:
  - Still slower than centralized ECDSA (100-150ms faster for single signer)
  - Security proof uses simulation-based definition requiring adversary see randomizer R before message—stronger than standard unforgeable-under-chosen-message-attack assumption
  - Requires all messages reliably broadcast to all parties (necessary for identification protocol)
  - 10% computational overhead compared to GG18 for identifiable abort capability
  - Pre-processing phase still required for efficiency; cannot be fully online

- Open questions / future work:
  - Whether to remove network versioning and integrate with other threshold protocols
  - Extending identifiable abort to other threshold cryptographic schemes
  - Optimal trade-offs between round complexity, bandwidth, and identification overhead

## Impact on our project

- Adopt/adapt? (yes/no) — notes:
  - **YES - HIGH PRIORITY**: If project involves distributed signature schemes, this protocol directly addresses failure detection in threshold ECDSA. The identifiable abort mechanism is state-of-the-art for practical deployment.
  - Recommended for: multi-sig authentication, decentralized key management, cryptocurrency wallets, distributed key splitting scenarios
  - Implementation note: Authors provide Go implementation based on Binance's tss-lib; code builds on well-established library ecosystem

- Pitfalls to avoid:
  - Do not use anonymous abort version (Section 5) if party attribution is required for accountability
  - Ensure all parties have reliable broadcast channels; protocol requires delivery of every message for identification to work
  - Be aware of ~10% computational overhead; not suitable for extremely latency-sensitive applications
  - Recognize that simulation-based security is stronger assumption than EU-CMA; ensure this matches your threat model

- Changes to evaluation / scope:
  - Benchmark DoS resistance: compare abort rate and identification latency vs GG18
  - Test identification protocol effectiveness with high numbers of participants (paper shows 2-10 participants)
  - Evaluate network efficiency gains/losses from broadcasting additional commitments
  - Assess practical deployment in scenarios with unreliable or hostile network conditions

## Practical notes

- Implementation hints / libraries mentioned:
  - Built on Binance's tss-lib (https://github.com/binance-chain/tss-lib) - high-quality production code
  - Uses Go implementation; single-threaded for benchmarking (though highly parallelizable)
  - Key dependency: elliptic curve library for ECDSA, Paillier encryption scheme, range proofs
  - Zero-knowledge proof generation is most parallelizable component (run separately per player)
  - Range proofs based on proofs from [28] (Gennaro-Goldfeder 2019 eprint)

- Datasets / benchmarks used:
  - Tested on 2.3 GHz Intel Core i5 with 16GB RAM
  - Benchmarks averaged over 50 runs
  - Measured single-threaded wall-clock computation time (excludes network latency)
  - Compared against GG18 protocol using same baseline

## Short quote

> "A key problem with all known threshold ECDSA protocols is that in the case of aborts, it is not possible to always identify which party is responsible for causing the signature to fail. In this section, we will show how to identify aborts. First of all we assume that all messages transferred between players are signed, so that it is possible to determine their origin. The protocol will abort in case any player deviates from the protocol in a clearly identifiable way by not complying with the protocol instructions—e.g. not sending a message when required."
