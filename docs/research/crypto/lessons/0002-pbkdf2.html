PBKDF2: Deriving Keys from Passwords

How PBKDF2 turns a low-entropy password into a high-quality cryptographic key, why iteration count matters, and exactly what role it plays in protecting your Ed25519 private key.

1. The Problem PBKDF2 Solves

Passwords are chosen by humans. They tend to be short, follow predictable patterns, and come from a small space — maybe 40–60 bits of entropy for a strong password, far less for a typical one. Cryptographic keys, on the other hand, need to be uniformly random strings from a space of 2128 or 2256. You cannot use a password directly as an AES key. NIST SP 800-132 §1

Password-Based Key Derivation Functions (PBKDFs) bridge this gap. They take a low-entropy password and produce a high-quality derived key, while deliberately making the derivation computationally expensive to slow down offline search attacks.

In this project: when an approver authenticates, the system derives enc_key = PBKDF2(password, key_salt, 600000, 32) and uses that 256-bit key to decrypt the approver's stored Ed25519 private key. PBKDF2 is the only thing standing between a database theft and private key recovery — assuming the attacker does not know the password.

2. Formal Definition

PBKDF2 is specified in RFC 8018 §5.2 (PKCS #5 v2.1) and approved by NIST SP 800-132 §5.3. The function signature is: RFC 8018 §5.2

PBKDF2(PRF, P, S, c, dkLen) → DK
ParameterMeaningOur Value
PRFPseudorandom function — HMAC with an approved hashHMAC-SHA-256
PPassword (octet string)approver's password
SSalt (non-secret, random octet string)key_salt (≥128 bits)
cIteration count (positive integer)600,000
dkLenDesired output length in octets32 (256 bits)
DKDerived keyenc_key (AES-256 key)

NIST requires dkLen ≥ 112 bits and salt randomly generated to ≥ 128 bits. NIST SP 800-132 §5.1, §5.3 Our 256-bit output and (assumed) ≥128-bit salt both conform.

3. The Algorithm in Detail

PBKDF2 divides the desired output into blocks, computing each block via a function F. RFC 8018 §5.2

Step 1 — Block decomposition

Compute l = ⌈dkLen / hLen⌉ blocks, where hLen is the PRF output length (32 bytes for HMAC-SHA-256). For our 32-byte output with a 32-byte PRF, l = 1.

Step 2 — The F function (one block)

Each block Ti is computed as the XOR of c successive PRF outputs, called iterates:

F(P, S, c, i) = U₁ ⊕ U₂ ⊕ ... ⊕ U_c

Where the iterates chain recursively:

U₁ = PRF(P, S || INT(i)) -- seed: salt + block index U₂ = PRF(P, U₁) -- each iterate feeds the next U₃ = PRF(P, U₂) ... U_c = PRF(P, U_{c-1})

Step 3 — Concatenate and truncate

DK = T₁ ‖ T₂ ‖ ... ‖ T_l <0..r-1>
Why this design? RFC 8018 describes it as "belt-and-suspenders":
  • The recursive chain (Uj = PRF(P, Uj-1)) means an attacker cannot compute iterate k without having computed all prior iterates. This removes parallelism within a single password guess: the chain is sequential.
  • The XOR of all iterates prevents the chain from degenerating to a small cycle of repeated values — if the chain happened to cycle, XOR mixing would still produce unpredictable output.

Visually, for our case (1 block, 600k iterations):

Password ──┐ │ key_salt ──┤──► PRF ──► PRF ──► PRF ──► ... (600,000×) ──► PRF ‖ INT(1) │ U₁ U₂ U₃ U_c │ ↓⊕ ↓⊕ ↓⊕ ↓⊕ │ T₁ T₁ T₁ ........ T₁ → DK

Each arrow represents one HMAC-SHA-256 call. The password is the HMAC key for all of them.

4. The Salt

The salt is a non-secret, randomly generated value stored alongside the encrypted private key. It is not a secret. Its purpose is to prevent precomputation attacks. RFC 8018 §4.1 NIST SP 800-132 §5.1, A.2.1

Without a salt, an attacker could precompute PBKDF2(password, …) for every word in a dictionary once and check all users in the database simultaneously. With a unique 128-bit salt per user, the attacker must redo all 600,000 iterations separately for each password candidate per user.

Possible keys per password = 2^sLen (where sLen is salt length in bits) With 128-bit salt: 2^128 possible (salt, password) combinations per password. An attacker cannot reuse a precomputed table across users or across key_salt values.

NIST salt requirements

  • Randomly generated portion of salt shall be ≥ 128 bits
  • Generated using an approved Random Bit Generator (SP 800-90)
  • An optional purpose prefix may be prepended: S = purpose ‖ rv

RFC 8018 sets a lower floor of 64 bits; NIST's 128-bit requirement supersedes it for conforming implementations. NIST SP 800-132 §5.1

5. The Iteration Count and Why 600,000

An attacker who steals the database knows the salt and can evaluate PBKDF2 offline, trying candidate passwords one by one. The iteration count C directly multiplies their cost. NIST SP 800-132 A.2.2

Attack cost without PBKDF: 2^t evaluations (t = bits of password entropy) Attack cost with PBKDF2: C × 2^t evaluations

The iteration count is a linear work factor applied to both the attacker and the legitimate user. The legitimate user runs PBKDF2 once per login — a sub-second delay is acceptable. The attacker must run PBKDF2 once per guess, and they have billions of guesses to try.

Calibrating to hardware

NIST says: "The iteration count shall be selected as large as possible, as long as the time required to generate the key using the entered password is acceptable for the users." The minimum is 1,000. For especially critical keys, 10,000,000 may be appropriate. NIST SP 800-132 §5.2

Iteration countNIST statusOWASP 2024 guidance
1,000MinimumDeprecated — far too low for modern hardware
100,000AcceptableMinimum acceptable for PBKDF2-HMAC-SHA256
600,000Well within rangeOWASP recommended default (2023)
10,000,000Appropriate for critical keysHigh-security applications

Our choice of 600,000 aligns with the OWASP recommendation for PBKDF2-HMAC-SHA-256 for password hashing, applied here to the more sensitive task of key derivation. On a modern server CPU, 600,000 iterations of HMAC-SHA-256 takes roughly 300–500 ms — noticeable but acceptable for an infrequent signing operation.

The GPU problem: C is a CPU-time multiplier, but attackers can use GPUs or ASICs to run many PBKDF2 evaluations in parallel across different passwords (not within a single evaluation — the chain prevents that). A high-end GPU cluster can evaluate hundreds of thousands of PBKDF2-HMAC-SHA-256 chains per second. This is the key weakness compared to memory-hard functions. See Section 6.

6. Security Model and Limitations

What PBKDF2 provides

PBKDF2's security rests on HMAC being a Pseudorandom Function (PRF). If HMAC-SHA-256 is a PRF (argued from SHA-256's properties, proven under the assumption that the underlying compression function is a PRF), then PBKDF2's output is computationally indistinguishable from a uniformly random string of the same length — to any adversary who does not know P.

This is a weaker foundation than Ed25519's ECDLP-based proof. PBKDF2 does not have a tight formal reduction from a clean hardness assumption. NIST SP 800-132 provides an engineering argument (the iteration multiplier) rather than a cryptographic theorem. The security is genuine and widely accepted, but the proof chain is shorter.

What PBKDF2 does NOT provide

Memory-hardness. PBKDF2 can be computed with a tiny, fixed amount of RAM — one PRF call at a time. This allows attackers to build custom hardware (ASICs) or use GPUs, running many parallel password guesses cheaply.

Argon2id (RFC 9106, winner of the 2015 Password Hashing Competition) is memory-hard: it requires a configurable amount of RAM to evaluate, which cannot be reduced even with custom hardware. OWASP recommends Argon2id over PBKDF2 for new applications when FIPS compliance is not required.

Why PBKDF2 here anyway? PBKDF2-HMAC-SHA-256 is the only NIST-approved password-based KDF (NIST SP 800-132). Argon2id is not approved in FIPS 140-3. If the project ever needs to operate in a FIPS-validated context, PBKDF2 is the only option. This is the reason to document explicitly. See the ADR.

The security boundary

Attacker hasCan doPBKDF2's defense
DB only (key_salt, enc_private_key) Run offline dictionary attack against password 600,000 iterations multiplies cost by 600k
DB + cracks password Derive enc_key, decrypt private key, forge signatures None — password quality is the last defense
No DB access Nothing meaningful Not applicable

PBKDF2 raises the cost of cracking. It does not make cracking impossible. The actual last line of defense is password quality (entropy) and the bcrypt-verified login gate that prevents online guessing. PBKDF2 protects the offline case — after a DB breach.

7. In This Project

The complete key-derivation flow for an approver authentication event:

-- 1. Verify password against stored bcrypt hash (online gate) if not bcrypt.verify(password, password_hash): abort -- 2. Derive encryption key from password (this is PBKDF2's job) enc_key = PBKDF2( PRF = HMAC-SHA-256, P = password, -- approver's password S = key_salt, -- 128-bit random, stored in DB c = 600_000, -- OWASP-recommended iteration count dkLen = 32 -- 256 bits for AES-256 ) -- 3. Decrypt private key (AES-256-GCM, next lesson) private_key = AES_GCM_decrypt(enc_key, encrypted_private_key) -- 4. Sign and discard signature = Ed25519_sign(private_key, approval_record) zero_memory(enc_key, private_key)

PBKDF2 is evaluated once per approval event. The result (enc_key) is held in memory only for the duration of the decryption and signing operations, then zeroed. The key_salt is stored in the user_keys table alongside encrypted_private_key — both are public within the DB; neither needs to be secret.

Password change implication: If an approver changes their password, the PBKDF2-derived enc_key changes. This means the stored encrypted_private_key must be re-encrypted using the new enc_key at password-change time. Failing to do so makes the private key permanently unrecoverable. NIST SP 800-132 §5.4 explicitly requires this re-protection step.
