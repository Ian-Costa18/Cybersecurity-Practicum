Cryptography Lesson 0003

AES-256-GCM

Authenticated encryption — confidentiality and integrity in a single operation

1. The Problem GCM Solves

Encryption alone does not guarantee integrity. Counter mode (CTR) produces ciphertext that is malleable: flipping bit i of the ciphertext flips bit i of the plaintext upon decryption, with no error. An attacker who intercepts a ciphertext can alter it surgically without knowing the key.

A MAC alone guarantees integrity but not confidentiality. Running both separately requires two keys, two passes, and careful composition — a composition that is non-trivial to get right.

Galois/Counter Mode (GCM) is an Authenticated Encryption with Associated Data (AEAD) scheme. One key, one pass, one operation produces:

Confidentiality

Ciphertext is indistinguishable from random bytes under chosen-plaintext attack (IND-CPA).

Integrity + Authenticity

Authentication tag T detects any modification to the ciphertext, AAD, or IV.

From the spec
GCM can detect both accidental modifications of the data and intentional, unauthorized modifications. — NIST SP 800-38D, §3

2. Inputs and Outputs

The authenticated encryption function GCM-AEK takes three inputs and produces two outputs:

GCM-AEK(IV, P, A) → (C, T) Inputs: K — encryption key (256 bits for AES-256) IV — initialization vector (nonce); recommended 96 bits P — plaintext (the data to encrypt) A — additional authenticated data (AAD); authenticated but NOT encrypted Outputs: C — ciphertext; same bit length as P T — authentication tag; 128 bits (recommended)

AAD is the key concept that distinguishes AEAD from plain encryption. Data in A is covered by the tag — any modification to it will cause decryption to return FAIL — but it is transmitted in the clear. This is useful for binding context to a ciphertext: for example, a user ID or record version that must be authentic but need not be secret.

"The plaintext and the AAD are the two categories of data that GCM protects. GCM protects the authenticity of the plaintext and the AAD; GCM also protects the confidentiality of the plaintext, while the AAD is left in the clear." — SP 800-38D §5.2.1

Inputs must satisfy these length constraints (SP 800-38D §5.2.1):

len(P) ≤ 239 − 256 bits (~64 GiB per invocation) len(A) ≤ 264 − 1 bits 1 ≤ len(IV) ≤ 264 − 1 bits (96 bits strongly recommended) tag length t ∈ {128, 120, 112, 104, 96, 64, 32}

3. Two Building Blocks

GCM is built from two sub-algorithms. Understanding each is necessary to understand the security proof.

3.1 GCTR — Counter Mode Encryption

GCTR is standard CTR mode using the underlying block cipher. It takes an Initial Counter Block (ICB) and a bit string X of arbitrary length:

ICB → [inc32] → CB2 → [inc32] → CB3 → ··· ↓ ↓ ↓ [CIPHK] [CIPHK] [CIPHK] ↓ ↓ ↓ X1 → [⊕] → Y1 X2 → [⊕] → Y2 X3 → [⊕] → Y3

Each counter block is encrypted with AES, then XORed with the corresponding plaintext block. Only the forward direction of AES is used — decryption re-runs the same operation. The 32 least-significant bits of the counter are incremented per block; the upper 96 bits remain fixed.

3.2 GHASH — Polynomial Hash over GF(2128)

GHASH is a keyed hash function built on multiplication in the binary Galois (finite) field GF(2128). The field uses the reduction polynomial corresponding to R = 11100001 || 0120.

Given a hash subkey H and a sequence of 128-bit blocks X1, ..., Xm:

GHASHH(X1 ‖ X2 ‖ ... ‖ Xm): Y0 = 0128 Yi = (Yi-1 ⊕ Xi) • H for i = 1, ..., m return Ym In closed form: X1•Hm ⊕ X2•Hm-1 ⊕ ... ⊕ Xm•H

This is a polynomial in H, evaluated over GF(2128). The security of GHASH as an authentication function follows from this structure: a non-zero polynomial of degree ≤ m over GF(2128) has at most m roots, so the probability that two distinct messages produce the same GHASH output is bounded by m/2128.

Lemma 2 — McGrew & Viega
GHASH is [l/w+1]2−l-almost XOR-universal when inputs are restricted to total length ≤ l bits. This is the formal basis for GCM's authentication security. — IACR ePrint 2004/193, Appendix A

4. The Hash Subkey H

Before any encryption, GCM derives a secret called the hash subkey H:

H = AESK(0128) — encrypt the 128-bit zero block with the GCM key

H is the key for the GHASH function. It is derived from K (not independent of it), which means:

  • H must never be revealed — any leak of H completely breaks authentication.
  • H changes every time a new key K is generated, which is why key freshness is explicitly required (SP 800-38D §8.1).
Critical
SP 800-38D §5.3 states that all intermediate values in GCM — including H — shall be kept secret. An implementation that uses H as an unpredictable value or integrity check value in any other context breaks this requirement.

5. GCM-AE: The Full Algorithm

Algorithm 4 from SP 800-38D §7.1:

Algorithm GCM-AEK(IV, P, A) — Authenticated Encryption
  1. H = AESK(0128) — derive hash subkey from the zero block
  2. Compute J0 from IV:
      If len(IV) = 96: J0 = IV ‖ 031 ‖ 1
      Otherwise: J0 = GHASHH(IV ‖ 0s+64 ‖ [len(IV)]64) — for non-standard IV lengths
  3. C = GCTRK(inc32(J0), P) — CTR-mode encryption; J0 is incremented once before use as ICB
  4. Compute padding: u = 128·⌈len(C)/128⌉ − len(C), v = 128·⌈len(A)/128⌉ − len(A)
  5. S = GHASHH(A ‖ 0v ‖ C ‖ 0u ‖ [len(A)]64 ‖ [len(C)]64) — authenticate AAD, ciphertext, and their lengths
  6. T = MSBt(GCTRK(J0, S)) — encrypt the GHASH output to form the tag; J0 (not incremented) is the ICB here
  7. Return (C, T)

Two things to note about J0. In step 2, for the standard 96-bit IV, it is formed by appending 0311 — a 31-bit zero string followed by a 1 bit. This makes J0 the block whose 32-bit counter field equals 1. In step 3, inc32(J0) is used as the starting ICB for plaintext encryption — so the first counter used for encryption is counter value 2. In step 6, J0 itself (counter value 1) encrypts the GHASH output S to produce the tag. This ensures the counter blocks used for plaintext and the counter block used for the tag are always disjoint.

6. GCM-AD: Decryption and Tag Verification

Algorithm 5 from SP 800-38D §7.2. The authenticated decryption function accepts (IV, C, A, T) and returns either the plaintext P or the special error code FAIL:

Algorithm GCM-ADK(IV, C, A, T) — Authenticated Decryption
  1. Check that len(IV), len(C), len(A), len(T) are all supported; if len(T) ≠ t, return FAIL
  2. H = AESK(0128)
  3. Compute J0 from IV (same as GCM-AE step 2)
  4. P = GCTRK(inc32(J0), C) — decrypt ciphertext
  5. Compute padding u, v
  6. S = GHASHH(A ‖ 0v ‖ C ‖ 0u ‖ [len(A)]64 ‖ [len(C)]64)
  7. T' = MSBt(GCTRK(J0, S)) — recompute expected tag
  8. If T = T', return P; else return FAIL
Implementation requirement
Plaintext P MUST NOT be released until tag verification succeeds (step 8). Releasing plaintext before verification — even partially — destroys the AEAD guarantee. The decryption oracle returns P or FAIL as a single atomic output.

7. The IV Uniqueness Requirement

SP 800-38D §8 states the uniqueness requirement in bold:

Requirement — SP 800-38D §8
The probability that the authenticated encryption function will ever be invoked with the same IV and the same key on two or more distinct sets of input data shall be no greater than 2−32.

What Happens When an IV Is Reused

This failure mode is described in SP 800-38D Appendix A and analyzed formally in McGrew & Viega §5:

  1. If the same (K, IV) pair is used for two encryptions, both ciphertexts are produced by XORing with the same keystream. An attacker who observes both can XOR them to cancel the keystream and learn P1 ⊕ P2.
  2. More critically: from the two (ciphertext, tag) pairs produced under the same (K, IV), an attacker can solve for the hash subkey H algebraically. This is a linear equation over GF(2128).
  3. Once H is known, the attacker can construct a valid authentication tag for any (IV, ciphertext, AAD) of their choosing. Authentication is entirely broken.
  4. With authentication broken, the attacker can control the plaintext produced by GCM-AD — the malleability of CTR mode is fully exploited.
Consequence
IV reuse in GCM does not "weaken" security — it eliminates both confidentiality and authenticity in a way that is undetectable without access to the key. In practice this means IV reuse is catastrophic in the same category as key exposure.

Approved IV Constructions (SP 800-38D §8.2)

Deterministic (§8.2.1)

IV = fixed_field ‖ invocation_field. The invocation field is a counter that increments with each encryption. Requires persistent counter storage. Supports unlimited invocations if the counter field is 64 bits.

RBG-based (§8.2.2)

IV is generated by a CSPRNG. Random field must be ≥ 96 bits. Total invocations capped at 232 per key to keep IV collision probability ≤ 2−32. Simpler to implement correctly.

For our use case (encrypting one key share per user authentication event), RBG-based with a 96-bit random IV is the appropriate choice. The per-key invocation count stays well below 232.

8. Security Model

GCM security rests on a single assumption. From McGrew & Viega §4:

Security Foundation
"The security of GCM stands on a single cryptographic conjecture: the block cipher E is assumed to be a secure pseudorandom permutation (PRP)." — McGrew & Viega, §4

This is a tighter and cleaner basis than PBKDF2. Two formal theorems are proved:

Theorem 1 — Confidentiality

Any adversary that can distinguish GCM encryption from a random function can also distinguish AES from a random permutation with essentially the same advantage.

Theorem 2 — Authentication

Any adversary that can forge a valid ciphertext/tag pair can distinguish AES from a random permutation. Forgery advantage is bounded by the AES-PRP advantage plus small correction terms.

Instantiated for AES-256-GCM with 96-bit IV, 128-bit tag, and fewer than 248 invocations per key (Corollary 1 of McGrew & Viega):

Distinguishing advantage ≤ AAES-256 + q2·2−116 − q·2−89.4 Forgery advantage ≤ AAES-256 + q2·2−116 − q·2−89.4 − q·2−128 Where q = number of authenticated encryption queries (invocations per key) AAES-256 = distinguishing advantage against AES-256 from a random permutation (believed negligible)

For a concrete example: if AES-256 is indistinguishable from a random permutation and fewer than 248 invocations are processed per key, the attacker advantage is no more than approximately 2−18.

Contrast with PBKDF2: PBKDF2 has no tight formal proof from a single hardness assumption — its security argument is that the iteration count multiplies attacker cost. GCM has a formal reduction: if you break GCM, you break AES. This makes the PBKDF2 → AES-256-GCM layering sound: the key derivation step produces a key; the encryption step is formally secure if that key is secret.

9. Application: DPK Encryption

In the multi-party approval proxy, AES-256-GCM is used to encrypt each user's threshold key share (Distributed Private Key, DPK). The enc_key from PBKDF2 (lesson 0002) is the GCM key K.

Inputs: K = enc_key — 256-bit key derived by PBKDF2-HMAC-SHA256(password, salt, 600000) IV = random_bytes(12) — 96-bit IV from CSPRNG; unique per encryption event P = dpk_bytes — the Ed25519 key share (32 bytes) A = user_id ‖ version — bound to the ciphertext; authenticated but not encrypted Outputs stored in database: (ciphertext, tag, iv) — 32 bytes + 16 bytes + 12 bytes = 60 bytes total

The AAD binding is critical. Including user_id and a schema version in AAD means a ciphertext produced for user A cannot be replayed as a valid ciphertext for user B — GCM-AD will return FAIL. This prevents cross-account key confusion even if an attacker can move ciphertext rows between users.

Design decision
Each successful authentication event that re-encrypts the DPK must generate a fresh 96-bit IV from the system CSPRNG. The IV is stored alongside the ciphertext. Key rotation (new password) derives a new enc_key and re-encrypts from scratch with a new IV — the same requirement as NIST SP 800-132 §5.4.
