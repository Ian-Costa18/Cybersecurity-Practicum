Cryptography · Lesson 01

Ed25519: What It Is, Why It's Secure, and Why We Use It

From the original paper through to the formal EUF-CMA proof

Why this matters for the project: Every approval decision in the proxy is signed with an Ed25519 private key. For docs/cryptography.md and the ADR to be defensible, you need to be able to state what security property Ed25519 provides, what assumption it reduces to, and why it was chosen over RSA and P-256 ECDSA.

1. The Formal Structure of a Signature Scheme

A signature scheme is formally a triple of algorithms (KGen, Sign, Vfy):

Definition 1 — Signature Scheme (Brendel et al., §III-A)

KGen(pp) takes public parameters and outputs a key pair (pk, sk).

Sign(sk, m) takes a secret key and message; outputs a signature σ.

Vfy(pk, m, σ) deterministically outputs 1 (valid) or 0 (invalid).

Correctness: for all honestly-generated key pairs and all messages, Vfy(pk, m, Sign(sk, m)) = 1.

Ed25519 is a concrete instantiation of this triple, built from the EdDSA (Edwards-curve Digital Signature Algorithm) construction over the twisted Edwards curve birationally equivalent to Curve25519.

2. The Concrete Algorithms

Parameters

The following parameters are fixed for Ed25519 (Bernstein et al. 2011, §2; Brendel et al. 2021, Table II):

The curve equation is: −x² + y² = 1 + dx²y² (twisted Edwards form). This form has a complete addition law — the group operation has no exceptional cases, which enables constant-time implementation without special-casing.

Key Generation

# Sample a random 256-bit secret key
k ←$ {0,1}²⁵⁶

# Expand it with SHA-512
h = SHA-512(k)        # 512 bits: h[0..255] and h[256..511]

# Clamp: set bit 254, clear bits 0-2
a = 2²⁵⁴ + Σ_{3≤i≤253} 2ⁱ·h[i]

# Public key: scalar multiplication of base point
A = a·B               # A is a point on E(F_q); encoded as 32 bytes

return (pk=A, sk=k)

The clamping step (setting bit 254, clearing bits 0–2) ensures a is a multiple of the cofactor 8, which places A in the prime-order subgroup and prevents small-subgroup leakage attacks.

Signing

# Derive deterministic nonce from private key seed + message
r = SHA-512(h[256..511] ‖ m)   # integer in [0, 2⁵¹² − 1]

# Nonce commitment
R = r·B

# Challenge hash (binds R, public key, and message)
c = SHA-512(R ‖ A ‖ m) mod ℓ

# Scalar response
S = (r + c·a) mod ℓ

return σ = (R, S)     # 32 + 32 = 64 bytes
Critical property: deterministic nonce
r is derived by hashing a long-term secret seed with the message — not drawn from an RNG. ECDSA requires a fresh random nonce per signature; if that nonce is ever reused across two messages, the private key is immediately recoverable. This is exactly what happened to Sony's PlayStation 3 in 2010, exposing their long-term code-signing key. Ed25519 makes nonce reuse structurally impossible.

Verification

# Parse signature
Parse σ as (R, S)
# Verify R, A are on the curve E
Check R, A ∈ E

# Recompute challenge
c = SHA-512(R ‖ A ‖ m) mod ℓ

# Cofactor-cleared group equation
Accept iff:  8·S·B = 8·R + 8·c·A

The multiplication by 8 (the cofactor) clears any small-subgroup component, making verification correct even when inputs are not strictly in the prime-order subgroup.

3. The Security Goal: EUF-CMA

Definition 2 — EUF-CMA (Brendel et al., §III-A)

A signature scheme is EUF-CMA secure (Existentially Unforgeable under Chosen Message Attack) if no probabilistic polynomial-time adversary can, after seeing the public key and obtaining signatures on any messages of its choice, produce a valid signature on a new message it never asked to sign — except with negligible probability.

This is the minimal requirement for a signature scheme to be useful. In the approval proxy context: even an attacker who has observed many legitimate approvals cannot fabricate a valid approval record for a decision that never happened.

A stronger property, SUF-CMA (Strong Unforgeability), additionally prevents producing any new (message, signature) pair — even for a message that was previously signed. This prevents signature malleability (the ability to produce a second valid signature for the same message).

4. The Security Proof Chain

Brendel et al. (2021) provide the first detailed formal proofs. The proof chain for Ed25519-IETF (the variant standardised in RFC 8032, which adds the check S ∈ {0,...,ℓ−1}) is:

ECDLP hard on E(F_q)
↓ birational equivalence preserves ECDLP difficulty (Bernstein et al. §2)
ECDLP hard on Curve25519
↓ CID identification protocol is IMP-KOA secure (Theorem 1)
CID is IMP-PA secure (Theorem 2)
↓ CID is HVZK — commitments recoverable from public key, challenge, response (Lemma 1)
Fiat-Shamir transform applied to IMP-PA-secure CID
↓ Standard Fiat-Shamir result in the random oracle model
Ed25519-Original is EUF-CMA secure (Theorem 3)
↓ S ∈ {0,...,ℓ−1} bounds check eliminates malleability
Ed25519-IETF is SUF-CMA secure (Theorem 4)

The hardness assumption at the base: the Elliptic Curve Discrete Logarithm Problem (ECDLP). Given a point A = a·B, recover a. No sub-exponential algorithm is known for the Ed25519 curve. The best known attacks (Pollard's rho) cost approximately 2¹²⁸ operations — the stated security level.

Key insight: the proof gap
The original Bernstein et al. paper (2011) did not contain formal proofs. Security was conjectured from the Schnorr lineage. Brendel et al. (2021) filled this gap and also revealed a subtlety: key clamping means not every element of E(F_q) is a valid public key, causing the reduction to occasionally abort. This introduces a small, negligible tightness loss — the reduction is slightly less efficient than in pure Schnorr, but the security bound remains strong at Ed25519's parameters.

For the approval proxy, SUF-CMA is the right target. The S ∈ {0,...,ℓ−1} check ensures no adversary can produce a second valid signature for an approval record that was already signed — preventing any ambiguity in the audit log. Any standard library implementing RFC 8032 will include this check.

6. Why Ed25519 Over the Alternatives

From Bernstein et al. (2011), benchmarked on the same hardware (Intel Westmere):

SchemeSign (cycles)Verify (cycles)Public keySignature
Ed2551987,548273,36432 bytes64 bytes
ECDSA P-2561,790,9362,087,50064 bytes64 bytes
RSA-307231,456,912231,536384 bytes384 bytes

Beyond performance, the structural differences matter:

  • vs ECDSA P-256: ECDSA requires a fresh random nonce per signature. Nonce reuse immediately reveals the private key (proven by Nguyen & Shparlinski, 2003). Ed25519's deterministic nonce eliminates this risk entirely. ECDSA also requires modular inverse computation; Ed25519 does not.
  • vs RSA-3072: RSA verification is faster, but signing is ~360× slower, and keys/signatures are 6–12× larger. RSA's security relies on integer factorisation; ECDLP on Curve25519 achieves the same 128-bit security level with dramatically smaller parameters.
  • Side-channel resistance: The twisted Edwards addition law is complete — no exceptional cases depending on the input points. This means the addition formula always executes the same sequence of operations, with no secret-dependent branching. P-256 ECDSA implementations require additional countermeasures to avoid timing leakage; Ed25519 achieves this by construction.

7. How This Applies to the Approval Proxy

Project context

In the proxy, Ed25519 serves one function: cryptographically binding an approval decision to an approver's identity.

At authentication time: approval_signature = Ed25519Sign(private_key, canonical_json(approval_record))

For audit: Ed25519Vfy(public_key, canonical_json(approval_record), approval_signature) — no password, no approver cooperation required.

The security properties map directly to the threat model:

  • EUF-CMA → An attacker with DB access but no password cannot forge an approval, even after observing many legitimate signatures. (If the password is also cracked, the private key can be decrypted — that attack is addressed by PBKDF2's cost, not Ed25519.)
  • SUF-CMA → An attacker cannot produce an alternative valid signature for an existing approval record (no ambiguity in the audit log).
  • Deterministic nonce → The proxy never needs per-signing randomness; no risk of nonce reuse even under a faulty RNG.
  • Persistent public key → Password changes only re-encrypt the private key. The public key is unchanged; all past signatures remain verifiable permanently.

Check Your Understanding

Five questions. Each is grounded in the paper content above. Ask the agent if anything is unclear.

1. An attacker has read access to the proxy database but does not know any approver's password. They see encrypted_private_key, public_key, and many stored (approval_record, approval_signature) pairs. What specifically prevents them from forging a new approval signature?

2. Why does Ed25519 use a cofactor of 8, and what does the multiplication by 8 in the verification equation 8·S·B = 8·R + 8·c·A accomplish?

3. Ed25519-Original does not achieve SUF-CMA. What practical consequence would this have if the approval proxy used a malleable variant?

4. What single parameter change distinguishes Ed25519-IETF from Ed25519-Original, and what security property does it unlock?

5. The approval proxy's authentication flow decrypts the private key transiently in memory, signs, then discards it. What property of Ed25519's signing algorithm makes this safe — specifically, why does it not require the private key to persist between signing operations for different approvals?
