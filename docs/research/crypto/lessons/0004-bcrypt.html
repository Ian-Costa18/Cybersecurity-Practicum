bcrypt: Construction and Security

Source: Provos & Mazières, "A Future-Adaptable Password Scheme," USENIX FREENIX 1999. Read in full.

1. The Problem with Fixed-Cost Password Hashing

In 1976, the UNIX crypt(3) function could hash fewer than 4 passwords per second on a VAX. By 1999, a modern Alpha workstation with optimized software could compute over 200,000 crypt operations per second. An attacker can now hash entire dictionaries of common passwords and compare the results to any stolen password file.

The core flaw: crypt and MD5 crypt both have fixed computational cost. No matter how fast hardware gets, the function stays equally cheap. The only long-term protection against offline guessing is a password hashing function whose cost can be tuned upward as hardware improves.

Provos and Mazières formalize this insight and present bcrypt, a password hashing function with provably adaptable cost.

2. Formal Definition: ε-Secure Password Functions (§3)

Let s be the user's secret password drawn from distribution D. Let t be any additional non-secret input (e.g., a salt). The predictability of distribution D is:

R(D) = maxs∈D Pr(s)

The probability of the most likely single password. High predictability = weak passwords.

An attacker is modeled as a randomized boolean circuit A with |A| gates (measuring computational work). The function F(s,t) is an ε-secure password function if both conditions hold:

  1. Partial information hardness: Knowing F(s,t) gives an attacker no meaningful advantage in learning any predicate of s — advantage over guessing is bounded by (ε/2)·|A|·R(D).
  2. Second preimage hardness: Finding s' ≠ s with F(s,t) = F(s',t) is as hard as guessing — advantage bounded by ε·|A|·R(D).

Both conditions scale with |A| (attacker's work) and R(D) (password predictability). A stronger password distribution (lower R(D)) gives the attacker less leverage even for a fixed ε.

No formal proof for bcrypt. The paper states explicitly: "we cannot formally prove bcrypt ε-secure, any flaw would likely deal a serious blow to the well-studied blowfish encryption algorithm." The security argument is heuristic — it reduces to Blowfish's empirical strength. Contrast with AES-256-GCM, which has tight formal reductions to the AES PRP assumption.

Three required properties

  • One-way with large salt: Second preimage resistance + salt space large enough to defeat precomputation (rainbow tables). Salt must be random and unique per password.
  • Adaptable cost: Cost must be tunable upward as hardware improves, without changing the stored verifier format.
  • Efficient on general CPUs; resistant to specialized hardware: Minimize the advantage attackers gain from custom chips, bitslicing, or GPU parallelism.

3. Eksblowfish: The Cost-Parameterizable Key Schedule (§4)

Blowfish (Schneier 1993) is a 64-bit block cipher using a 16-round Feistel network. Its state consists of:

  • 18 32-bit P-array subkeys: P1, ..., P18
  • 4 S-boxes S1–S4, each with 256 32-bit words (4 KB total)

The Feistel round function is:

F(a, b, c, d) = ((S₁[a] ⊕ S₂[b]) + S₃[c]) ⊕ S₄[d]    (addition mod 2³²)

Eksblowfish ("expensive key schedule blowfish") adds a cost parameter and a 128-bit salt to Blowfish's key schedule. The key innovation is that cost enters as an exponent — doubling cost adds exactly one bit to the cost parameter.

EksBlowfishSetup(cost, salt, key)
  state ← InitState() // fill P-array and S-boxes with digits of π
  state ← ExpandKey(state, salt, key)
  repeat 2cost times:
    state ← ExpandKey(state, 0, salt)
    state ← ExpandKey(state, 0, key)
  return state

ExpandKey(state, salt, key) XORs the key cyclically into the P-array, then uses the 128-bit salt as a 16-byte key to Blowfish-encrypt successive 64-bit blocks, replacing the P-array entries in pairs, then continuing through all S-box entries.

After the initial ExpandKey(state, salt, key), the function alternately re-expands with zero-padded salt and zero-padded key for 2cost iterations. The entire subsequent state depends on both salt and key — no part of the computation can be precomputed without knowing both.

4. The Bcrypt Algorithm (§5, Figure 3)

bcrypt(cost, salt, pwd)
  state ← EksBlowfishSetup(cost, salt, pwd)
  ctext ← "OrpheanBeholderScryDoubt" // 192-bit magic value
  repeat 64 times:
    ctext ← EncryptECB(state, ctext) // encrypt all three 64-bit blocks
  return Concatenate(cost, salt, ctext)

The 64-round encryption loop uses the expensive Eksblowfish state derived from the password and salt. Almost all of bcrypt's time is spent in EksBlowfishSetup — the 64-round loop is cheap by comparison.

The output format (used in OpenBSD and virtually all modern implementations):

$2a$NN$22-char-base64-salt31-char-base64-hash

Example: $2a$12$R9h/cIPz0gi.URNNX3kh2OPST9/PgBkqquzi.Ss7KIUgO2t0jWMUW
  • $2a$ — bcrypt version identifier (distinguishes from DES crypt, MD5 crypt)
  • NN — cost parameter (log₂ of iterations: cost=12 → 4,096 iterations)
  • 22 base64 characters — the 128-bit random salt
  • 31 base64 characters — the 192-bit ctext output

The 128-bit salt space is large enough that even in a 15,000-entry password file, the expected number of salt collisions is negligible (Section 6.2.1 — requires 241 salts for uniqueness in that file size; 2128 far exceeds this).

5. Why Bcrypt Resists Bitslicing and Hardware Attacks (§6.2.3)

Bitslicing is a software technique discovered by Biham that eliminates DES's costly bit-transpositions by treating a 64-bit processor as 64 parallel 1-bit processors. On a 300 MHz Alpha, bitsliced DES achieves ~5× speedup. "John the Ripper" (a password cracker) uses this to devastating effect against DES crypt.

Bitslicing works because DES S-boxes are fixed and well-known — they can be hardcoded into the boolean circuit.

Bcrypt S-boxes are different:

  • Each S-box entry is computed dynamically from the password and salt during EksBlowfishSetup
  • The 4 KB of S-box state changes continuously throughout the algorithm
  • Every (password, salt) pair produces different S-boxes
  • S-boxes must be stored separately for every simultaneous execution — no sharing

Consequence: Bitslicing cannot be applied to bcrypt. An attacker cracking passwords in parallel cannot amortize the S-box computation across attempts. Each guess requires its own full key schedule.

Additionally, because bcrypt relies on CPU-efficient operations (bitwise XOR, modular addition, memory lookups into L1 cache), custom ASIC hardware provides little speedup advantage over general-purpose CPUs — a property explicitly designed in per Section 3.

6. Cost Scaling Over Time

The cost parameter is a log₂ exponent: cost=10 → 1,024 iterations; cost=11 → 2,048; cost=12 → 4,096. Each increment doubles the computation time. At publication (1999) the defaults were:

  • cost=6 for normal users (feasible verification time; 240,000 crypt/s hardware benchmark from the paper)
  • cost=8 for the superuser

Modern hardware guidance (2025): cost=12 is a common minimum for interactive logins; cost=14 for higher-security contexts. The point is that operators can raise cost without changing the stored hash format — just re-hash on next login.

7. Comparison with DES Crypt and MD5 Crypt (§6.1)

DES crypt

  • 12-bit salt (4,096 unique values) weak
  • Fixed: 25 DES encryptions fixed cost
  • Bitslicing applies vulnerable
  • Max 8-char password

MD5 crypt

  • 12–48 bit salt better
  • Fixed cost fixed cost
  • Significant cracker optimizations exist
  • No password length limit

bcrypt

  • 128-bit salt strong
  • 2cost adaptable iterations adaptable
  • Bitslicing blocked resistant
  • Max 72-byte password

Neither DES crypt nor MD5 crypt support an adaptable cost — their protection against offline attack erodes monotonically as hardware improves. Bcrypt's protection holds as long as the operator raises the cost factor to match hardware.

8. Bcrypt vs Argon2id

Argon2id (winner of the Password Hashing Competition, 2015) adds memory-hardness: the function requires a configurable amount of RAM, which dramatically increases the cost of GPU and ASIC-based attacks (GPUs have limited per-core memory bandwidth). Bcrypt has no memory-hard property.

For the approval proxy, the ADR justifying bcrypt over Argon2id rests on:

  • Bcrypt is in passlib and virtually every web framework — library availability is excellent
  • The proxy is a low-throughput system; the number of login attempts is small, making GPU farms less relevant
  • Increasing the cost factor compensates for the absence of memory-hardness in the proxy's threat model
  • Bcrypt has two decades of production deployment and cryptanalysis — risk of unknown implementation bugs is lower

If the threat model changes (e.g., high-value target, stolen database, attacker with GPU cluster), migrating to Argon2id would be the correct decision.

9. Role in the Approval Proxy

bcrypt is used only for login password verification.

When a user sets their password, the system stores bcrypt(cost, random_salt, password). At login, the system re-derives the hash and compares. No key material is derived from this output.

PBKDF2 is used for key derivation: enc_key = PBKDF2(password, salt, 600k, SHA-256) to encrypt the Ed25519 key share. These are distinct primitives with distinct roles. Confusing them (e.g., trying to derive a key from bcrypt output) would be an implementation error.
