# ADR 0005: Ed25519 over RSA-2048 and P-256 ECDSA for Approval Signing

## Status

Accepted

## Context

The approval proxy requires each approver to produce a cryptographic signature binding their identity to an approval record at authentication time. The signature must be verifiable later using only a stored public key, with no approver cooperation required (see [ADR 0004](0004-asymmetric-key-approval-signing.md)). The required security property is **SUF-CMA** (strong unforgeability under chosen-message attack): an adversary with signing oracle access cannot produce any new valid signature, including alternative signatures on previously signed messages.

Three signing scheme families were evaluated.

### Option A: RSA-PSS with 2048-bit keys

RSA-PSS achieves EUF-CMA (standard unforgeability, not strong) under the RSA problem assumption (factoring) in the random oracle model. PSS-RSA signatures are deterministic given a fixed salt; this eliminates nonce randomness as an attack surface.

- **Key size:** 2048 bits (256-byte public key, 256-byte signature)
- **Security assumption:** Integer Factorization Problem (IFP) — factoring n = pq
- **Unforgeability:** EUF-CMA; no standard proof of SUF-CMA for RSA-PSS
- **Concern:** Signatures are 256 bytes versus Ed25519's 64 bytes. The IFP assumption, while well-studied, is computationally harder to reduce cleanly to a decision problem compared to ECDLP. Key generation requires reliable large-prime generation, which is implementation-sensitive.

### Option B: P-256 ECDSA (NIST curve)

ECDSA on the NIST P-256 curve achieves EUF-CMA under the ECDLP assumption in the generic group model. The standard ECDSA construction requires a fresh random nonce per signature.

- **Key size:** 256 bits (32-byte public key, up to 72 bytes DER-encoded signature)
- **Security assumption:** ECDLP on NIST P-256
- **Unforgeability:** EUF-CMA; signature malleability is a known property — SUF-CMA is not achieved without additional constraints
- **Concern 1 (nonce reuse):** ECDSA requires a fresh random k per signature. Nonce reuse k = k' for two messages reveals the private key via d = (s − s')^{-1}(r(k' − k)) mod n. This is the attack that recovered the PS3's private signing key. It requires a cryptographically strong RNG per signature operation.
- **Concern 2 (biased nonces):** Lattice attacks (e.g., Minerva, 2019) recover private keys from ECDSA signatures generated with biased nonces — a property of some hardware RNG implementations. RFC 6979 deterministic ECDSA eliminates this, but it is an opt-in that implementations may omit.
- **Concern 3 (curve provenance):** NIST P-256's seed was generated via an unverifiable process. No practical attack is known, but the curve is not "verifiably random" in the sense that its parameters cannot be confirmed to be free of special structure known to its designers.
- **Concern 4 (malleability):** ECDSA signature (r, s) is malleable to (r, −s mod n), both verifying under the same public key. This prevents SUF-CMA without an explicit non-malleability constraint in the verification algorithm — which the standard does not include.

### Option C: Ed25519-IETF (RFC 8032)

Ed25519 is EdDSA over the twisted Edwards curve birationally equivalent to Curve25519 (p = 2^255 − 19, a = −1). The nonce r = SHA-512(sk_nonce ‖ M) mod l is derived deterministically from the signing key and message — no per-signature randomness is required.

- **Key size:** 256 bits (32-byte public key, 64-byte signature)
- **Security assumption:** ECDLP on Curve25519
- **Unforgeability:** The Ed25519-IETF variant (RFC 8032), which includes the check S ∈ {0,...,l−1} in verification, achieves **SUF-CMA** under ECDLP (Brendel et al. IEEE S&P 2021, Theorem 4). This closes the malleability gap present in the original Ed25519 variant, which achieves only EUF-CMA (Theorem 3).
- **Deterministic nonces:** Eliminates all RNG-dependent attack surfaces. Nonce reuse is structurally impossible — the same (key, message) pair always produces the same signature.
- **Curve provenance:** Curve25519 parameters are derived from the prime p = 2^255 − 19 by explicit criteria (Montgomery B smallest positive integer such that (B−2)/4 is small; verifiably random). No unverifiable seed process.

## Decision

**Chosen: Ed25519-IETF (RFC 8032, Option C)**

## Rationale

1. **SUF-CMA is achieved with a formal proof.** Brendel et al. (2021) Theorem 4 formally proves that Ed25519-IETF achieves SUF-CMA under ECDLP on Curve25519 in the random oracle model. RSA-PSS achieves EUF-CMA only. ECDSA achieves EUF-CMA only (malleable without additional constraints). SUF-CMA is the correct property for non-repudiable approval records — an approver must not be able to claim an alternative signature on the same record.

2. **Deterministic signatures eliminate RNG dependency.** Ed25519 derives its nonce from a hash of the private key and the message. No per-signature random input is required. ECDSA's random nonce requirement is a significant implementation liability (PS3 attack, Minerva); RFC 6979 mitigates it but is opt-in and not universally implemented.

3. **128-bit classical security with a compact key.** Curve25519 provides approximately 128-bit security against the best known generic ECDLP algorithms (Pollard's rho). This matches AES-256 at 256-bit key / 128-bit security. RSA-2048 provides approximately 112-bit security (NIST SP 800-57 Table 2); a stronger RSA key (3072 bits) would be needed to match.

4. **Verifiably random curve parameters.** Curve25519's parameters are derived by explicit criteria and do not require trusting the process that generated a seed. P-256's parameters require trusting NIST's undocumented seed.

5. **Compact output.** Ed25519 signatures are 64 bytes; RSA-2048 signatures are 256 bytes; P-256 DER-encoded signatures are up to 72 bytes. The compact representation simplifies storage, encoding, and comparison in audit records.

6. **Ubiquitous, constant-time implementations.** libsodium, OpenSSL (≥1.1.1), Python's `cryptography` library, and Go's `crypto/ed25519` all provide constant-time Ed25519 implementations. Constant-time RSA and ECDSA are harder to verify in third-party code and historically more prone to timing side-channels.

## Implications

- Key generation uses Ed25519 as specified in RFC 8032 §5.1.5.
- Signing uses the Ed25519-IETF variant; verification includes the S ∈ {0,...,l−1} bounds check.
- The private key scalar is the 256-bit output of the clamping operation on SHA-512(seed)[0..255].
- All approval records are signed; signatures are stored in plaintext alongside the record and the approver's public key.

## Trade-offs Accepted

- **Ed25519 is not FIPS 140-2 approved in all implementations.** FIPS 186-4 did not include Ed25519; FIPS 186-5 (2023) adds it. If FIPS compliance is required before library support catches up, this requires re-evaluation.
- **No formal proof for non-IETF variants.** Only the Ed25519-IETF variant (with S bounds check) achieves SUF-CMA. Implementations that omit this check achieve only EUF-CMA. The implementation must enforce the IETF variant.
